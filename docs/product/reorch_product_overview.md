# ReOrch 智策产品说明书

版本：v0.1
当前阶段：MVP 已完成，正在合作实验室进行初步试用和验证
产品定位：面向复杂离散制造的异常调度决策操作层

## 1. 产品一句话

ReOrch 智策是一套部署在 ERP、MES、APS 之上的异常调度决策系统。当车间执行中的计划被设备故障、急单、物料延期、质量返工等异常打断时，系统帮助计划员快速完成影响分析、候选方案生成、多目标评估、人工确认、受控回写和经验沉淀。

它不是一个让大模型直接接管排产的工具，而是把 AI、规则引擎、排程求解器、质量门、数字孪生和人工审批组织成一条可验证、可审计、可追责的异常响应流程。

## 2. 要解决的问题

复杂离散制造企业通常并不缺少系统。ERP 管订单和主数据，MES 管执行和现场反馈，APS 生成计划和排程，Excel、电话、会议和即时通讯承担大量临时协调。但在真实生产现场，最消耗组织效率的往往不是“首次排不出计划”，而是计划执行后不断被异常打断。

典型问题包括：

- 关键设备故障后，计划员需要判断哪些工单受影响、哪些订单会延期、是否应等待维修或马上局部重排。
- 急单插入后，计划员需要平衡交付优先级、已有承诺、换线成本和资源占用。
- 物料延期或质量返工发生后，生产、计划、采购、销售之间需要快速形成一致口径。
- 计划员往往依赖个人经验手工试排，决策过程难追溯，结果难复盘，新人难接手。
- 现有系统可以记录事实和执行计划，但异常发生后的“判断、比较、解释、确认、沉淀”链路常常不完整。

ReOrch 的核心判断是：制造企业真正需要的不是另一个孤立排程工具，而是一层面向异常响应的决策操作层。它把异常发生后的关键动作产品化，让企业从“靠人临时协调”转向“系统生成候选、计划员确认、过程可审计、经验可复用”。

补充内容：这类产品不适合用“全自动排产”作为主叙事。工业调度的生产责任不能交给自由生成式模型。更专业的产品边界是：系统提高决策速度和方案质量，计划员保留最终确认权，生产写回必须经过权限、预览、审计和回滚机制。

## 3. 目标用户

### 3.1 核心用户

| 用户角色 | 主要诉求 | ReOrch 提供的能力 |
| --- | --- | --- |
| 计划员 | 快速知道异常影响哪些订单，拿到可比较方案，减少手工试排 | 影响分析、Top-K 候选方案、多目标对比、推荐解释 |
| 生产主管 | 评估重排是否会造成现场混乱，控制扰动和执行风险 | 扰动范围、换线次数、资源切换、质量门风险提示 |
| 设备/维修负责人 | 说明故障恢复时间和设备可用性对计划的影响 | 异常接入、维修时长字段、资源状态和影响范围 |
| 供应链/物料协调 | 判断物料延期或齐套风险对排程的影响 | 数据接入、物料可用性约束、后续扩展的规则候选 |
| IT/系统集成负责人 | 需要低侵入接入现有 ERP/MES/APS，并保证权限和审计 | 标准 API、只读接入、回写适配、审计记录 |
| 业务负责人 | 关心延期、加班、换线、停机等待和协调成本能否下降 | ROI 指标、异常处理时间、采纳率、执行反馈 |

### 3.2 首批适配客户

ReOrch 更适合以下客户：

- 复杂离散制造企业，而不是稳定流水线或连续流程制造。
- 多品种小批量，订单优先级变化频繁。
- 工艺路线复杂，设备能力、前后工序、换线和资源日历会影响排程。
- 每周有多次会实质影响计划的异常事件。
- 已有 ERP/MES/APS，或至少能导出工单、工序、设备、当前排程和异常日志。
- 计划员仍大量依赖 Excel、会议和个人经验做异常重排。
- 管理层愿意用历史异常 replay、shadow mode 和计划员并行验证来衡量价值。

暂不优先的客户：

- 异常频率低、排程变化少、生产节拍稳定的场景。
- 基础数据严重缺失，连工单、工序、设备、交期和当前排程都无法重建的场景。
- 客户希望首期就替换 ERP/MES/APS 全部主系统能力。
- 客户要求无人值守自动写回生产系统，但尚未建立权限、审计和回滚机制。

## 4. 产品定位

ReOrch 的定位不是 ERP、MES 或 APS 的替代品，而是它们之上的异常响应层和经验资产层。

| 层级 | 现有系统通常负责 | ReOrch 负责 |
| --- | --- | --- |
| ERP | 订单、物料、客户、财务和主数据 | 读取订单、交期、优先级等决策字段 |
| MES | 生产执行、设备状态、工序报工、现场事件 | 接收异常事件和执行反馈 |
| APS | 基准排程、有限产能计划、全局计划优化 | 提供排程快照或作为回写对象 |
| ReOrch | 不是主数据和执行底座 | 异常影响分析、候选方案比较、人工确认、审计和案例沉淀 |

更具体地说，ReOrch 解决的是“计划被打断以后怎么办”。系统首先锁定异常发生时的排程快照，再计算影响范围和交期风险，然后根据异常类型、资源状态和业务目标生成多套候选方案，最后通过质量门和多目标评价把方案交给计划员确认。

补充内容：如果客户现有 APS 已经能在几分钟内完成影响分析、Top-K 候选方案、解释、人工确认、审计和执行反馈闭环，ReOrch 不应作为替换方案销售。更合理的价值点应转向轻量异常入口、解释层、案例资产层或与现有 APS 的插件式集成。

## 5. 核心产品闭环

ReOrch 的产品闭环由十个步骤构成。

### 5.1 异常接入

异常可以来自三类入口：

- 人工文本输入：计划员或现场人员输入“CNC-02 设备故障，预计维修三个小时，急单可能延期”。
- 结构化 MES/IoT 事件：外部系统直接提交设备、异常类型、发生时间、预计恢复时间、来源和资源信息。
- 历史 replay 或数字孪生样例：用于验证系统是否能在已知异常上复现影响范围和候选方案。

自然语言输入不会直接进入求解器。系统会先由 Incident Agent 做异常理解，抽取资源 ID、维修时长、异常类型和风险提示，并给出置信度。只有高置信且求解器支持的异常，才会转换为标准 Incident 并进入决策流。

### 5.2 标准 Incident

标准 Incident 是系统后续所有判断的入口。它至少包含：

- 异常 ID。
- 异常类型。
- 发生时间。
- 资源 ID。
- 来源系统。
- 严重等级。
- 描述。
- 原始 payload。
- 幂等键和审计字段。

这一层的价值在于把混乱的现场表达转换成稳定的数据契约。无论异常来自文本、MES、IoT 还是人工表单，进入影响分析前都必须变成同一套结构。

### 5.3 严重等级判断

系统目前有两层严重等级。

第一层发生在异常接入中心，依据资源关键性、是否瓶颈、是否有冗余、活跃工单数量等信息进行初始分级：

| 等级 | 判断逻辑 |
| --- | --- |
| P1-Critical | 瓶颈设备，或高风险配置 |
| P2-High | 关键资源，且影响 3 个以上活跃工单 |
| P3-Medium | 一般资源，影响 1-2 个活跃工单 |
| P4-Low | 非关键资源，且有冗余备份 |

第二层发生在影响分析之后。如果发现交期 Breach 风险，系统只升级不降级：

```text
P4 -> P3
P3 -> P2
P2 -> P1
P1 保持 P1
```

前端影响分析区会展示“严重等级依据”，包括初始等级、生效等级、分级规则、关键因子、Breach 工单数和影响分析结论。这个设计的目的不是让等级看起来更复杂，而是让计划员知道系统为什么把一次异常判为 P1/P2/P3/P4，避免黑盒标签。

### 5.4 排程快照锁定

异常决策必须基于某一个明确的排程状态。系统会读取或生成 ScheduleSnapshot，记录异常发生时的工单、工序、设备、开始结束时间、前后约束和资源分配。

如果没有排程快照，系统只能接入异常，不能可靠生成候选方案。此时系统应降级提示，而不是伪造重排结果。

### 5.5 影响分析

影响分析引擎计算：

- 直接受影响工序。
- 下游间接受影响工序。
- 受影响工单。
- 受影响资源。
- 预估延迟分钟数。
- 交期风险分布：Safe、Warning、Breach。
- 是否需要升级严重等级。

这一层必须是可计算、可追溯的。计划员不应只看到“风险高”，还要能看到是哪道工序、哪个工单、哪个资源、哪个交期被影响。

### 5.6 策略选择

系统根据影响范围、维修时间、受影响工单数量、交期风险和业务目标选择策略。当前策略包括：

- Wait and Repair：等待维修完成，尽量少扰动计划。
- Local Repair：局部修复，只调整受影响区域。
- Global Reschedule：全局重排，适合影响范围大或交期风险高的情况。

策略不是最终方案，而是指导求解器生成候选方案的方向。计划员仍然可以比较不同策略下的候选结果。

### 5.7 Top-K 候选方案生成

系统不会只给一个“系统最优答案”。工业调度通常不是单目标问题，不同方案之间存在明显取舍：

- 交期更稳，但调整工序更多。
- 改动更少，但关键订单风险更高。
- 瓶颈利用率更好，但换线次数增加。
- 总完工时间更短，但现场执行复杂度更高。

因此，ReOrch 生成 Top-K 候选方案，让计划员看到可选择空间。候选方案进入推荐前必须通过硬约束和质量门。

### 5.8 质量门与多目标评价

质量门负责阻断明显不可用或高风险方案：

- 资源能力不匹配。
- 工序前后顺序冲突。
- 同一设备同一时间重叠。
- 冻结工序被移动。
- 数据缺失导致方案不可审计。
- 回写权限或幂等风险不满足要求。

多目标评价负责比较可用方案：

- 交付风险。
- 总延期分钟。
- 关键订单 OTD。
- 调整工序数量。
- 换线和资源切换次数。
- 瓶颈资源利用率。
- 执行复杂度。
- 方案置信度。

质量门回答“能不能进入确认”，多目标评价回答“哪一个更适合当前业务目标”。

### 5.9 推荐解释与人工确认

系统会基于结构化 KPI 和质量门结果解释推荐原因。解释必须基于已有事实，不允许模型重新编造指标或隐藏风险。

计划员可以：

- 接受推荐方案。
- 接受但做局部调整。
- 拒绝并选择其他方案。
- 填写 override 原因。

人工确认不是形式化按钮，而是生产责任边界。没有确认，就不应进入真实生产回写。

### 5.10 回写准备、执行跟踪和案例沉淀

确认后，系统生成回写预览或回写指令，用于对接 MES/APS。当前阶段应优先采用 mock writeback、人工审核回写或 dry-run，不建议直接开放无人值守写回。

执行后，系统记录：

- 采纳方案。
- 人工调整内容。
- override 原因。
- 回写结果。
- 实际执行偏差。
- 最终延期、换线、加班和协调成本。

这些数据会进入案例库和偏好学习模块。长期看，ReOrch 不只是一次性生成方案，而是逐步沉淀企业的异常处理资产。

## 6. 功能模块

### 6.1 异常接入中心

能力：

- 接收 MES、IoT、人工上报或自然语言文本。
- 校验来源是否合法。
- 校验必填字段。
- 做 10 分钟窗口内的同资源去重。
- 生成严重等级和分级证据。
- 发布事件流或写入本地运行态。

价值：

- 减少重复报警。
- 统一异常数据结构。
- 将严重等级从“主观标签”变成可解释规则。

### 6.2 Incident Agent

能力：

- 解析自然语言异常文本。
- 抽取设备/资源 ID。
- 解析维修时长。
- 识别风险提示。
- 输出置信度。
- 低置信度时进入人工确认。

边界：

- 不直接触发求解器。
- 不决定最终方案。
- 不绕过标准 Incident。

### 6.3 Impact Analysis Agent

能力：

- 调用确定性影响分析引擎。
- 输出受影响工序、工单、资源和交期风险。
- 生成严重等级升级依据。
- 保留可审计 trace。

边界：

- 不修改排程。
- 不生成候选方案。
- 不自由编造影响数据。

### 6.4 Strategy Agent

能力：

- 根据影响范围、维修时长、风险分布和目标模式选择策略。
- 输出策略类型、置信度和关键因素。
- 支持后续结合历史案例和偏好画像。

边界：

- 策略只是求解方向，不等于生产计划。
- 低置信度策略需要计划员重点复核。

### 6.5 Solver Agent

能力：

- 调用求解器和策略编排层。
- 生成多套候选方案。
- 保留候选方案的策略来源和目标信息。

边界：

- 不允许自由生成排程。
- 必须调用确定性算法、约束校验和质量门。

### 6.6 Evaluation Agent

能力：

- 对候选方案做多目标评价。
- 支持不同目标模式：交付优先、稳定优先、瓶颈优先、成本优先、平衡模式。
- 输出推荐方案和候选方案比较矩阵。

边界：

- 不替代计划员确认。
- 不隐藏劣势指标。

### 6.7 Explanation Agent

能力：

- 将候选方案差异解释为计划员能理解的业务语言。
- 说明推荐方案为何在当前目标下更合理。
- 暴露风险、trade-off 和替代方案。

边界：

- 只能解释结构化指标和方案差异。
- 不得修改评价结果。

### 6.8 Feedback Agent

能力：

- 将计划员的拒绝理由、覆盖原因和现场反馈结构化。
- 生成规则候选。
- 将人工经验沉淀为案例资产。

边界：

- 规则候选必须人工审核。
- 单次 override 不应直接升级为生产硬约束。

### 6.9 Case Memory 与 Preference Learning

能力：

- 存储历史异常、影响范围、候选方案、采纳方案和执行结果。
- 检索相似案例。
- 学习计划员和车间在交付、稳定、成本之间的真实偏好。
- 为后续策略选择和方案排序提供辅助信号。

边界：

- 样本不足时只能观察，不应用于自动调整全局策略。
- 偏好学习不能覆盖硬约束。

## 7. 前端工作台

当前前端工作台围绕计划员的异常决策流程组织。

### 7.1 左侧异常事件列表

提供：

- 异常列表。
- AI 文本接入。
- MES/IoT 结构化事件接入。
- 加载演示场景。
- 异常类型、严重等级、状态、时间筛选。

产品价值：

- 把“异常发生”变成决策流入口。
- 支持人工文本和系统事件两种真实来源。
- 让沙盒演示、实验室试用和未来系统接入走同一条标准链路。

### 7.2 中间影响分析区

提供：

- 异常基础信息。
- 影响范围统计。
- 交期风险分布。
- 严重等级依据。
- 受影响工单表。
- 策略推荐。
- 事件时间线。

产品价值：

- 让计划员先理解影响，再比较方案。
- 避免直接把用户推到复杂排程结果里。
- 把“为什么严重”“为什么升级”“为什么不升级”显式展示。

### 7.3 候选方案与推荐确认

提供：

- Top-K 候选方案。
- 多目标评价。
- 推荐方案。
- 质量门结果。
- 人工确认。
- override 记录。

产品价值：

- 将计划员的真实权衡显式化。
- 不把系统推荐包装成唯一答案。
- 为后续案例沉淀和偏好学习提供结构化样本。

## 8. AI 的角色边界

ReOrch 中的 AI 负责提高理解、解释和沉淀效率，但不承担生产最终责任。

AI 可以做：

- 将自然语言异常转成结构化字段。
- 将现场规则和 override 原因转成候选规则。
- 将方案差异解释成业务语言。
- 将历史决策沉淀成案例标签和偏好信号。
- 在低风险环节提高交互效率。

AI 不应做：

- 直接生成最终生产排程。
- 绕过求解器和硬约束校验。
- 绕过人工确认写回生产系统。
- 把未验证的现场经验直接发布为硬规则。
- 把实验室或数字孪生结果宣称为客户生产收益。

这种边界看似保守，但对工业场景是必要的。高风险 AI 产品的关键不是模型更“主动”，而是系统能持续证明：每个结论来自哪里、经过了哪些检查、谁做了确认、出了问题如何追责和回滚。

## 9. 数据契约

### 9.1 最小必需数据

客户要跑通异常重决策，至少需要提供：

- 工单列表。
- 工序列表。
- 工序标准时长。
- 工序前后约束。
- 设备/资源清单。
- 设备能力标签。
- 当前排程快照。
- 工单交期。
- 当前执行状态。
- 异常事件。

### 9.2 强建议数据

为提升方案可用性，建议补充：

- 设备日历和班次。
- 维护窗口。
- 物料齐套时间。
- 换线规则和换线成本。
- 关键客户或关键订单标识。
- 人员技能矩阵。
- 工装、夹具、刀具约束。
- 历史异常日志。
- 历史计划版本和实际执行结果。

### 9.3 数据质量门

如果关键字段缺失，系统应降级：

- 没有排程快照：只能接入异常，不能生成可靠候选方案。
- 没有工序前后约束：不能保证方案工艺可行。
- 没有设备能力：不能保证资源分配正确。
- 没有交期：无法计算交付风险。
- 没有执行状态：无法正确冻结已开始或已完成工序。

补充内容：客户数据不足时，产品交付不应硬做成“AI 重排效果验证”。更专业的最小交付物应是数据就绪评估、字段映射报告和异常决策可行性报告。

## 11. 当前阶段与上线边界

当前产品已经完成 MVP，具备端到端异常决策闭环和可互动前端。现阶段正在合作实验室进行初步试用和验证，重点验证：

- 流程是否符合计划员真实工作方式。
- 异常接入是否足够自然。
- 影响分析是否能帮助计划员快速判断风险。
- Top-K 方案是否覆盖可接受选择。
- 质量门和风险提示是否足够清晰。
- override 和案例沉淀是否能被计划员接受。

当前可支持：

- 合作实验室试用。
- 内部演示。
- 只读数据验证。
- 历史异常 replay。
- 数字孪生 shadow 代理验证。
- 人工确认 dry-run。
- mock writeback 或人工审核回写预览。

当前不建议：

- 直接接入客户生产系统并开放自动写回。
- 无人值守自动调度。
- 用当前数字孪生结果直接承诺客户现场 ROI。
- 在未完成回滚、审计、权限和 SLA 验收前生产上线。

结论：ReOrch 当前足够支持受控试用和小范围验证，但不应被描述为已经完成生产级无人值守上线。进入生产试点前，需要完成客户真实数据只读接入、历史异常 replay、shadow mode、计划员并行验证、回写演练、权限审计、失败回滚和运维验收。

## 12. 部署和集成路径

建议采用四阶段交付。

### 12.1 数据就绪评估

目标：

- 获取客户样例数据。
- 完成字段映射。
- 检查工单、工序、设备、交期、快照和异常日志是否可用。
- 输出数据质量问题和补齐清单。

交付物：

- 字段映射表。
- 数据质量报告。
- 异常决策可行性评估。

### 12.2 历史 replay

目标：

- 选取 10-30 条历史异常。
- 用历史排程快照复现影响分析。
- 对比系统候选方案和历史人工处理方式。
- 校准严重等级、风险阈值和目标权重。

交付物：

- replay 报告。
- 候选方案比较。
- 失败样本归因。
- 阈值校准建议。

### 12.3 Shadow mode

目标：

- 系统只读接入客户现场数据。
- 与计划员并行运行，不直接影响生产。
- 记录系统建议、计划员选择和实际执行结果。
- 验证采纳率、风险提示和业务指标。

交付物：

- shadow mode 周报。
- 方案采纳率。
- override 原因分析。
- 业务价值估算。

### 12.4 受控生产试点

目标：

- 小范围开放人工确认后的回写。
- 建立权限、幂等、防重复提交、失败回滚和审计导出。
- 只覆盖已验证的异常类型和车间范围。

交付物：

- 生产试点验收报告。
- 回写安全报告。
- 审计包。
- 运维和回滚预案。

## 13. 商业价值

ReOrch 的商业价值来自四类效率提升。

### 13.1 缩短异常响应时间

计划员不再从头查找受影响工序和订单。系统自动锁定快照、计算影响范围、生成候选方案，并把取舍展示出来。

### 13.2 降低交付风险

系统显式计算 Safe、Warning、Breach 风险，并把关键订单、总延期分钟和瓶颈资源影响放到同一评价框架中。

### 13.3 减少组织协调成本

异常处理过程从电话、会议和个人经验转为结构化流程。不同角色可以围绕同一组候选方案和指标讨论，而不是各自拿一份表。

### 13.4 沉淀企业异常处理资产

每一次选择、拒绝、调整和执行偏差都会成为案例。长期看，企业可以形成自己的策略模板、偏好画像和异常处理知识库，减少对少数资深计划员的依赖。

## 14. 与传统系统的关系

### 14.1 与 APS

APS 通常擅长基准排程、有限产能计划和全局优化。ReOrch 不替代 APS，而是面向异常发生后的快速决策和确认。理想状态下，ReOrch 可以读取 APS 快照，也可以把确认后的变更回写给 APS 或 MES。

### 14.2 与 MES

MES 是生产执行系统，记录现场状态、设备事件、工序报工和执行结果。ReOrch 可以接收 MES 事件，也可以把确认后的调整以受控方式传回 MES。

### 14.3 与 ERP

ERP 提供订单、客户、交期、物料和财务口径。ReOrch 不负责主数据治理，但需要读取关键字段用于交期风险和 ROI 评估。

### 14.4 与通用大模型平台

通用大模型平台提供自然语言理解、生成和 Agent 能力。ReOrch 的差异在于工业业务对象、排程求解器、质量门、人工确认、回写安全和案例资产闭环。通用模型可以作为组件，但不能替代产品系统本身。

## 15. 产品原则

1. 先可用，再智能。先保证异常接入、影响分析、候选方案和人工确认闭环，再谈更高阶的自动化。
2. 先只读，再回写。客户现场验证必须从只读和 shadow mode 开始。
3. 先单场景，再扩展。首期聚焦关键设备故障等高频异常，跑通后再扩展插单、物料和质量返工。
4. 先质量门，再推荐。任何方案进入推荐前都必须通过硬约束和风险检查。
5. 先解释取舍，再追求最优。计划员需要知道为什么推荐、代价是什么、替代方案是什么。
6. 先沉淀经验，再学习偏好。样本不足时不要过早让系统自动学习和调整策略。
7. 先承认边界，再承诺价值。客户信任来自可验证结果，而不是过度承诺。

## 17. 关键风险与产品应对

| 风险 | 表现 | 产品应对 |
| --- | --- | --- |
| 数据质量不足 | 无法重建排程或影响范围 | 先做数据就绪评估，不强行承诺重排效果 |
| 计划员不采纳 | 系统方案不符合现场隐性约束 | 记录 override，沉淀规则候选和偏好信号 |
| 扰动过大 | 方案理论可行但现场难执行 | 质量门标记扰动和执行复杂度，推荐低扰动替代 |
| 客户要求自动化过快 | 希望直接无人值守写回 | 坚持只读、shadow、人工确认、回写演练四阶段 |
| 与现有 APS 冲突 | 客户担心替换主系统 | 明确定位为异常响应层，优先读写接口和插件化 |
| AI 可信性不足 | 用户担心模型胡说 | 高风险环节不用 LLM，解释必须绑定结构化数据和 source refs |

## 18. 产品判断

ReOrch 的价值不在于把“AI 排产”说得更大，而在于把一个真实、频繁、昂贵但长期依赖人工经验的生产决策过程拆成可执行的产品闭环。

对复杂离散制造企业来说，异常永远不会消失。真正能形成长期价值的系统，不是承诺异常发生后自动给出唯一正确答案，而是持续帮助组织更快看清影响、更稳比较方案、更清楚解释取舍、更安全执行变更，并把每一次人工判断沉淀成下一次决策的资产。

这也是 ReOrch 当前最清晰的产品方向：先在关键设备故障等高频异常上证明受控决策闭环，再用客户真实数据、历史 replay 和 shadow mode 验证可复制价值，最后逐步扩展为企业级异常调度决策操作层。
