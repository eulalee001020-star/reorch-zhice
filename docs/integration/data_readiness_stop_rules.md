# Data Readiness 停损规则

## 为什么需要停损线

ReOrch 的价值不应建立在“客户给什么数据都能跑”的假设上。数据质量不足时，产品应该主动降级为 readiness 评估和字段修复，而不是输出看似合理但无法追溯的重排方案。

## 分级规则

| 分级 | 条件 | 系统允许做什么 | 系统不承诺什么 |
| --- | --- | --- | --- |
| Stop | blocker > 0 | 输出缺口、字段映射和修复建议 | 不生成重排方案，不进入 ROI 或采纳率统计 |
| Repair | readiness < 0.70 | 数据治理、mapping 修复、客户字段确认 | 不承诺调度效果 |
| Replay Only | 0.70 <= readiness < 0.85 | 历史 replay、人工复核、约束补充 | 不自动写回，不做正式价值归因 |
| Shadow Ready | readiness >= 0.85 且 blocker = 0 | shadow mode、Top-K 候选复核、采纳/驳回统计 | 仍不允许无人值守调度 |

## 当前产品入口

前端已新增独立页面：

```text
登录 -> 数据就绪
```

页面提供两个演示入口：

- `评估可运行样本`：展示可进入 shadow / replay 的数据状态。
- `评估缺字段样本`：展示 blocker 触发后的停损逻辑。

## 最低字段合同

客户至少需要提供：

- 工单：工单 ID、产品、数量/优先级、交期。
- 工序：工序 ID、所属工单、工时、可用资源、前后依赖。
- 资源：设备/工位 ID、能力、日历、状态。
- 当前排程：工序开始/结束时间、资源绑定。
- 异常事件：类型、发生时间、资源/工单对象、预计影响。
- 执行反馈：采纳方案、实际完成、延期、人工 override 原因。

缺少执行反馈时，系统可以做方案生成和 shadow 对比，但不能形成真实采纳率和 ROI 结论。
