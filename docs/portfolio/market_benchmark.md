# 市场需求与行业先进标准对标

## 1. 结论

ReOrch 智策的市场切口成立，但必须聚焦。合理的第一阶段不是“所有制造业智能排产”，而是“中大型离散制造企业的异常调度决策层”，尤其是汽车零部件、电子装配、精密机加工、装备制造、新能源设备等多品种小批量、高异常频率、有 ERP/MES 数据基础的场景。

如果把市场讲成“603 万制造业企业都能卖”，会显得不专业。更可信的说法是：先完成 MVP 和合作实验室试用，再从有数据基础、有异常频率、有交付压力、有计划员成本的单车间场景切入，用受控试点验证 ROI。

## 4. 行业先进标准对标

### 4.1 从自由 Agent 转向可控 Workflow

阿里云 Model Studio 文档中，Agent 更适合开放式对话和提示词驱动，Workflow 更适合固定、可重复、有明确输入输出的流程。工业调度属于高风险流程，因此 ReOrch 采用受控工作流：

```text
Incident -> Impact -> Strategy -> Solver -> Gate -> Confirm -> Writeback -> Case
```

这种设计比“让 Agent 自己决定下一步”更适合生产场景。

### 4.2 从模型效果转向业务验收

ReOrch 的验收指标不是 BLEU、ROUGE 或单纯准确率，而是：

- 异常到方案时间。
- 硬约束可行率。
- Top-N 覆盖历史人工策略。
- 方案采纳率。
- 延期分钟减少。
- 扰动和换线控制。
- 审计完整率。
- 回写幂等和失败处理。

### 4.3 从 MVP 到试点可落地

先进的 AI 产品作品集不应只展示漂亮界面，还要说明：

- 目标客户是谁。
- 第一个可付费场景是什么。
- 需要客户提供哪些数据。
- 如何接入而不破坏现有系统。
- 如何证明 ROI。
- 哪些能力尚未被验证。
- 数字孪生先行给出的 replay/shadow 代理证据是什么。

## 5. 竞争格局

成熟 APS/MOM/MES 厂商已经具备详细排程、有限产能、what-if 和动态重排能力。ReOrch 不能宣称“只有我们能重排”。

更合理的差异化：

| 方向 | 成熟 APS/MOM | ReOrch 切口 |
| --- | --- | --- |
| 目标 | 全厂级计划与执行协同 | 异常发生后的决策操作层 |
| 落地 | 主数据治理和系统实施较重 | 只读接入、历史 replay、shadow mode 起步 |
| 用户 | 计划、生产、供应链多个层级 | 先服务计划员和车间异常协调 |
| AI 价值 | 多为优化/预测/平台模块 | 异常理解、策略解释、经验沉淀、案例检索 |
| 验收 | 系统级计划能力 | 异常到方案时间、采纳率、扰动控制、ROI |

对外话术应避免“替代 APS”。更强的表达是“部署在 ERP/MES/APS 之上的异常响应层和经验资产层”。

## 6. 首批目标客户画像

优先：

- 中大型离散制造企业。
- 已有 ERP/MES，可能有 APS，但异常处理仍高度依赖人工。
- 每周至少 3-5 次对排程有实质影响的异常。
- 单车间可以量化计划员耗时、延期、换线、加班或停工等待。
- 愿意提供近 1-3 个月工单、工序、设备、日历、排程快照和异常日志。

暂不优先：

- 完全纸面化、数据基础不足的小厂。
- SKU 少、节拍稳定、异常影响低的流水线。
- 已有成熟 APS 且异常闭环已经非常成熟的头部工厂。
- 一上来要求全厂级计划、采购、库存、人力和财务一体化改造的项目。

## 7. 付费 PoC 设计

| 维度 | 设计 |
| --- | --- |
| 周期 | 4-6 周 |
| 范围 | 1 个车间、1 个高频异常类型、1 套只读数据源 |
| 数据 | 工单、工序、设备、日历、物料、基线排程、10-30 条历史异常 |
| 模式 | 只读接入 -> 历史 replay -> shadow mode -> 人工确认 dry-run |
| 验收 | 异常到方案时间、可行率、采纳率、延期减少、审计完整率 |
| 商业目标 | 20-30 万元付费 PoC，验证后转 30-50 万元/车间/年 |

当前项目阶段：MVP 已完成，正在合作实验室初步试用；数字孪生样例已先行形成 source refs、成本代理、replay/shadow 代理、阈值样本和审计包结构。客户现场试点时，应把这些代理证据替换为客户真实数据和计划员并行验证结果。

## 8. 主要风险与应对

| 风险 | 判断 | 应对 |
| --- | --- | --- |
| 数据质量差 | 工业 AI 最常见落地阻塞 | 先做 mapping validator 和 readiness report |
| 客户把 PoC 变成重咨询 | 范围失控会拖垮交付 | 限定单车间、单异常、单数据源 |
| AI 被误解为自动接管生产 | 高风险且不可信 | 明确 human-in-the-loop 和 writeback gate |
| 候选方案和人工偏好不一致 | 初期正常 | 用 override 和执行反馈沉淀客户私有策略 |
| Solver 性能不足 | 真实工厂规模可能更复杂 | 先 rolling window/local repair，再逐步扩展 |

## 9. 项目亮点

1. 有 AI 但不迷信 AI：知道 Agent、Workflow、求解器和人工责任的边界。
2. 有产品但不只做界面：覆盖数据接入、验收指标、风险控制、商业化路径。
3. 有 demo 但不伪装生产上线：明确 sandbox、数字孪生代理验证、合作实验室试用和客户现场 shadow mode 的差异。
4. 有行业切口：制造业智能化政策和企业数字底座成熟，但异常决策层仍存在真实缺口。
5. 有工程可信度：可运行系统、测试、CI、Docker Compose 和客户数据包，不是纯概念 PPT。
