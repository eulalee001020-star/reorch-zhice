# 服务领域 AI 迁移说明

## 1. 迁移判断

ReOrch 的核心资产不是某个排程算法界面，而是“高约束服务流程中的 AI 决策辅助框架”。制造异常调度、客服协同、售后工单、服务质检和客户成功运营都具有相似的产品结构：

```text
高频异常 / 高噪声输入
-> 数据与知识是否可信
-> 需要 AI 辅助理解与补全
-> 需要人工确认关键动作
-> 需要审计、指标和复盘
```

因此，ReOrch 的 Data Readiness、Evidence Stack、Agent Workflow、Guardrail、Human Confirmation 和 Case Memory 可以迁移到服务领域，但不能直接复制行业对象和指标，必须按服务流程重新建模。

## 2. 从工业异常到服务流程

| 工业异常调度 | 服务领域对应 |
| --- | --- |
| 设备故障、插单、物料延期 | 客诉、退款、故障报修、SLA 超时、升级请求 |
| 工单、工序、设备、排程快照 | 客户、订单、会话、服务工单、知识库、SLA 规则 |
| 计划员确认方案 | 坐席采纳、班长审核、服务运营确认 |
| 质量门 pass / warn / block | 回复风险、政策风险、合规风险、低置信升级 |
| 受控回写 MES/APS | 回写 CRM、工单系统、知识库或服务记录 |
| 案例沉淀 | 优质回复、失败样本、知识缺口、处理 SOP |

## 3. 迁移后应保留的产品原则

- AI 不直接替代一线人员做高风险承诺。
- 知识引用、客户上下文和操作记录必须可追溯。
- 低置信、敏感、越权、合规风险必须转人工或升级。
- 输出要可采纳、可驳回、可标注原因，而不是只给一次性答案。
- 上线后用采纳率、转人工率、错误率、SLA、质检结果和知识缺口持续校准。

## 4. 服务 AI 产品形态

### AI 客服 Copilot

```text
用户问题 -> 意图识别 -> 知识检索 -> 回复建议 -> 风险检查
-> 坐席确认 -> 客户回复 -> 采纳/修改/驳回记录 -> 知识缺口沉淀
```

适合解决回复慢、知识不一致、新人培训成本高和质检复盘困难的问题。

### 智能工单 Agent

```text
工单进入 -> 分类 -> 字段补全 -> SLA 风险 -> 派单建议
-> 人工确认 -> 处理结果 -> 重开/升级/关闭复盘
```

适合解决工单信息不完整、派单不稳定、SLA 风险发现晚和经验难复用的问题。

### 服务质检 Agent

```text
会话/工单记录 -> 规则匹配 -> 证据定位 -> 风险标注
-> 质检复核 -> 申诉处理 -> 规则与知识库迭代
```

适合解决人工抽检覆盖不足、质检口径不一致和问题归因不清的问题。

## 5. 迁移边界

ReOrch 能证明 AI 产品的框架能力，但服务领域落地仍需重新完成：

- 服务知识库与政策边界校准。
- 客服/运营/质检用户访谈。
- CRM、工单系统和知识库接口调研。
- 敏感场景、合规要求和话术审核。
- 上线后的 A/B、灰度、质检和反馈闭环。

结论：迁移不是换一套页面文案，而是复用“可控 AI 工作流 + 证据层 + 人工确认 + 指标闭环”的产品方法，重新建模服务行业对象、权限和验收指标。
