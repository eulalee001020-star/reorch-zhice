# 项目状态评估

本文从可信性、成本控制和商业价值三个维度评估 ReOrch 智策当前实现。项目已经完成 MVP，具备可互动演示和端到端异常决策闭环，目前正在合作实验室做初步试用和验证。当前可以支撑受控试用、只读验证和数字孪生 replay/shadow 代理验证，不建议直接作为生产系统上线。

## 1. LLM 结果是否可信可行

ReOrch 当前不是让 LLM 直接给排产答案，而是把 Agent 输出放进确定性验证链路。异常理解、策略组织和解释可以由 Agent 参与；影响计算、约束校验、候选方案、推荐排序和回写必须由结构化模型、求解器、质量门和人工确认控制。

| 判断项 | 当前状态 | 证据 | 提升空间 |
| --- | --- | --- | --- |
| 结构合法 | 已实现 | Pydantic models、TypeScript types、demo mapping validation | 接客户数据时补字段血缘和版本管理 |
| 数据可追溯 | MVP 已实现，数字孪生已补 source refs | Incident、ScheduleSnapshot、ImpactReport、AgentTraceStep、`validation_evidence.source_refs` 已串联 | 客户现场需要绑定真实工单、工序、设备、KPI 主键 |
| 硬约束 | 已实现核心版 | ConstraintValidator、HybridSolver、PlanRecommendationEngine、PlanQualityGate | BOM、人员、刀具、能耗等客户约束需要按现场数据补齐 |
| 业务风险 | 已实现核心版，数字孪生已给阈值样本 | risk_flags、quality_gates、recommendation_explanation、前端候选方案质量门标签、`threshold_calibration` | 风险阈值需要结合实验室反馈和客户历史异常校准 |
| 置信度 | 已实现 | Incident/Strategy/Recommendation confidence，低置信度不自动预选 | 需要记录人工覆盖后的置信度校准样本 |
| 审计 | 核心链路已实现，数字孪生已给审计包结构 | DecisionRecord、audit_metadata、writeback records、AgentTraceStep、`audit_package_proxy` | 生产级审计包导出格式、签名留痕和版本快照仍需增强 |

当前已补齐：异常决策主流程返回 `quality_gates`，并在候选方案比较表显示“质量门通过/预警/阻断”。这让质量门不只停留在文档和初始排产流程，而是进入核心异常工作台。

## 2. 成本与模型选择

当前默认 demo 没有开启外部 LLM 供应商，因此无 API Key 时仍不存在真实 token 账单。系统已经把低风险的异常理解、规则候选和计划员反馈结构化接入可配置真实 LLM Agent；配置 `LLM_ENABLED=true`、`LLM_API_KEY`、`LLM_BASE_URL` 和 `LLM_MODEL` 后，Agent trace 会记录模型、token、latency 和降级原因。数字孪生验证包中的 `model_cost_proxy` 仍作为成本计量结构，后续可替换为真实调用数据。

| 模块 | 当前实现 | 推荐模型/工具策略 | 成本判断标准 |
| --- | --- | --- | --- |
| 数据导入与字段映射 | 规则校验 + schema | 规则优先，小模型只做字段名候选 | schema error rate、人工修正字段数 |
| 异常文本理解 | 可配置真实 LLM Agent + 规则/正则降级 | 小模型或轻量分类器即可，低置信度人工确认 | 准确率、低置信召回、单次成本 |
| 影响分析 | 确定性引擎 | 不使用 LLM | 计算耗时、影响范围准确率 |
| 候选方案生成 | OR-Tools/求解器 | 不使用 LLM | 可行率、P95 求解耗时 |
| 方案解释 | 模板化解释层 | 中小模型可润色，但必须引用结构化证据 | 幻觉率、source_refs 覆盖率、人工采纳率 |
| 案例沉淀/反馈结构化 | 可配置真实 LLM Agent + 结构化抽取降级 | 小模型可用，输出必须 schema 校验 | 可复用规则比例、人工审核通过率 |

成本控制的原则是：能用规则和求解器解决的，不交给大模型；能用小模型稳定完成的，不调用大模型；模型只负责语义、总结、解释和经验抽取，不负责最终可行性判断。

[补充] 数字孪生阶段的成本结论是：核心决策链路可以先由确定性服务和求解器完成。低风险 Agent 可接真实 LLM 做语义增强，但必须保留确定性降级；进入客户现场试点后，再把 `model_cost_proxy` 升级为真实模型调用次数、input/output token、按模块分摊成本和 latency。

## 3. 商业化价值判断

项目价值不应靠“AI 排产”概念成立，而应靠异常响应速度、交付风险降低、计划员决策负担下降和经验资产沉淀成立。当前项目已经完成 MVP，并进入合作实验室初步试用；数字孪生先给出 replay/shadow 代理结果，后续再用实验室反馈和客户现场数据复核。

如果计划员没有采纳系统推荐，这不应被简单解释为系统失败或用户保守。更合理的做法是把不采纳样本拆成数据问题、策略/求解问题和组织流程问题：数据问题不进入 ROI 成功结论，策略问题用于调整权重和阈值，组织流程问题则沉淀为 override 原因、约束候选或审批规则。这样失败样本也能成为产品迭代证据。

| 维度 | 当前状态 | 证据 | 仍需验证 |
| --- | --- | --- | --- |
| 场景痛点 | 已定义清楚 | 设备故障等异常触发的重排决策 | 客户现场异常频率和损失分布 |
| 用户角色 | 已定义 | 计划员、主管、IT/集成、执行端 | 真实计划员工作流访谈 |
| 产品闭环 | MVP 已跑通，数字孪生已输出 replay/shadow 代理 | 异常 -> 影响 -> 方案 -> 推荐 -> 确认 -> 回写 -> 案例 | 现场 shadow mode 下与计划员实际方案对比 |
| 价值测算 | 已有工具和数字孪生样例 | ValueTrackingService、PoC Dashboard、`replay_shadow_proxy`：节省 82 分钟、延期减少 150 分钟、估算价值 7385 元 | 客户财务口径和实际单价 |
| 付费路径 | 已设计，待验证 | 先实验室试用，再单车间试点 | 愿付费访谈和采购阻力 |
| 风险边界 | 已明确 | 只读接入、人工确认、mock writeback | 真实系统适配和回写失败处置 |

判断项目是否具备真实业务价值，不能只看 demo 是否完整，而要看是否满足以下条件：

| 判断标准 | 当前结论 |
| --- | --- |
| 是否绑定高频、高损失、有人负责的业务事件 | 部分成立，需客户异常频率数据确认 |
| 是否比现有人工流程更快、更稳或更可追责 | 数字孪生代理指标成立，需现场 shadow mode 复核 |
| 是否能在不替换 APS/MES 的前提下上线 | 架构成立，需客户接口验证 |
| 是否有明确停损标准 | 已定义 Data Readiness 停损规则，并在前端提供独立入口 |
| 是否有愿付费主体和预算口径 | 方向成立，仍需真实客户访谈和报价验证 |

## 4. 上线判断

| 范围 | 当前判断 |
| --- | --- |
| 合作实验室试用 | 可以支持 |
| 内部演示和方案评估 | 可以支持 |
| 客户只读接入验证 | 可作为下一步推进 |
| 历史异常 replay / shadow mode | 可作为试点前置阶段 |
| 人工确认 dry-run 回写 | 有条件支持，需要客户 sandbox 和回滚演练 |
| 直接生产上线 | 不建议 |
| 无人值守自动调度 | 不支持 |
