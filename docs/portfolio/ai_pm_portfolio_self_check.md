# AI PM 作品集自查

## 结论

ReOrch 当前更适合定位为一个“深项目作品集”：用一个工业异常调度 MVP 展示真实问题判断、AI 适用性判断、Agent/Workflow 设计、模型不确定性控制、工程落地和数字孪生价值验证。

它不应该被包装成多个互不相关的浅项目，也不应该强行声称覆盖 RAG、多模态或微调。当前最强证据是受控 Agent 工作流、质量门、数字孪生验证包、可互动 demo、测试和上线边界判断。

## 1. 定位自查

| 标准 | 当前状态 | 判断 |
| --- | --- | --- |
| 是否从真实问题出发 | 从离散制造异常重排切入，用户是计划员、主管和集成/执行角色 | 通过 |
| 是否说明 AI 是否适用 | 明确 AI 负责异常理解、策略解释、规则候选和案例沉淀，求解和约束由确定性系统负责 | 通过 |
| 是否展示系统方案 | 有端到端流程、Agent 分工、质量门、确认和回写链路 | 通过 |
| 是否控制模型不确定性 | schema、硬约束、质量门、置信度、人机确认、审计 | 通过 |
| 是否推动工程落地 | FastAPI、React、OR-Tools、Docker Compose、CI、demo API、数字孪生接口 | 通过 |
| 是否用数据证明价值 | 数字孪生给出 82 分钟决策时间节省、150 分钟延期减少、7385 元单次估算价值 | 通过，但需实验室反馈复核 |

## 2. 项目数量策略

| 标准建议 | 当前选择 |
| --- | --- |
| 3-4 个高质量项目 | 当前采用 1 个深项目，不拆成伪项目 |
| RAG 知识库项目 | 不作为当前主项目；ReOrch 的核心不是知识问答 |
| Agent 工作流项目 | 是当前主项目核心能力 |
| 多模态/文档智能项目 | 当前未覆盖，不应硬包装 |
| AI 评测/商业化项目 | 通过数字孪生验证包、PoC Dashboard、ValueTrackingService 覆盖 |

判断：如果只投递一个项目，ReOrch 的深度已经足够支撑 AI PM 面试深讲；如果后续需要完整作品集组合，可以新增独立 RAG 或文档智能项目，而不是把当前项目强行改名拆分。

## 4. AI 能力映射

| 能力 | 当前证据 | 状态 |
| --- | --- | --- |
| 大模型理解 | 明确模型只做语义、解释、规则候选，不负责最终可行性 | 已体现 |
| RAG 理解 | 当前项目不以知识库问答为核心 | 不适用，不包装 |
| Agent 理解 | Incident、Impact、Strategy、Solver、Quality Gate、Explanation、Confirmation 分工清楚 | 已体现 |
| Embedding 理解 | 案例库和未来 pgvector 方向存在，但不是当前主证明点 | 部分体现 |
| 微调理解 | 当前没有微调需求，使用规则/求解器/结构化流程更合适 | 已说明取舍 |
| 上下文管理 | 通过 Incident、Snapshot、ImpactReport、CandidatePlan 传递状态 | 已体现 |
| 多模态理解 | 当前未覆盖图片/PDF/音频等复杂输入 | 不适用，不包装 |
| 评测能力 | `pytest`、前端构建、demo validation、digital twin validation evidence | 已体现 |
| 风险意识 | 幻觉、硬约束、回写、权限、审计、上线边界 | 已体现 |
| 工程协作 | API、类型模型、测试、CI、Docker、mock integration | 已体现 |
| 商业意识 | ROI、节省时间、延期减少、试点路径、上线判断 | 已体现 |

## 5. 避坑清单

| 常见错误 | 当前处理 |
| --- | --- |
| 只展示界面 | 文档强调问题定义、系统方案、质量门和验证证据 |
| 只说用了大模型 | 明确 AI 不是万能排产器，求解和约束由确定性系统负责 |
| 没有评测体系 | 已有测试、demo 校验、数字孪生验证包 |
| 没有失败案例 | 已有数字孪生风险样本，后续需补实验室失败样本库 |
| 不谈成本和延迟 | `model_cost_proxy` 当前显示外部 LLM 调用 0，后续升级 token telemetry |
| 不谈权限和合规 | 有 RBAC、人工确认、审计、上线就绪评估 |
| 过度包装结果 | 明确数字孪生是代理验证，客户现场仍需复核 |
