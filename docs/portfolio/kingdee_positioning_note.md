# 金蝶对标下的 ReOrch 定位说明

这份说明用于补充 ReOrch 的市场定位和能力边界。它不把金蝶写成一个被 ReOrch 替代的竞品，也不把 ReOrch 包装成新的 ERP/MES/MOM 主系统。更准确的说法是：在金蝶这类企业管理和制造运营平台已经具备完整底座能力的情况下，ReOrch 只验证一个更窄的问题是否成立——现场异常发生后，能不能用一套受控、可解释、可审计的决策层，帮助计划员更快完成判断。

## 现实参照

金蝶不是单点排程工具。公开资料显示，金蝶云星空智能制造解决方案覆盖计划排程、生产执行、仓储物流、质量、设备和能源等制造链路；金蝶 AI 星瀚/星瀚制造云覆盖计划、制造执行、设备、MES、WMS、QA/QC 和制造运营等能力；金蝶 AI 苍穹提供企业级 AI 平台、低代码、大模型和 Agent 架构能力。

这意味着 ReOrch 不能用“我们也能重排”作为核心卖点。成熟主系统已经能做很多计划、执行、协同和数据沉淀工作。ReOrch 更合理的位置，是贴在 ERP/MES/APS 之上的异常响应层：读取必要快照，生成可比较的候选方案，解释取舍，记录计划员选择和 override 原因，再把这次决策沉淀成可复盘资产。

| 维度 | 金蝶类主系统更擅长 | ReOrch 合理补位 |
| --- | --- | --- |
| 业务底座 | ERP/MES/MOM/PLM/WMS/设备管理等一体化流程 | 不替换底座，只读取异常决策需要的快照 |
| 计划与执行 | 全局计划、产供销协同、生产执行和组织权限 | 单车间、单高频异常下的 Top-K 决策辅助 |
| 数据治理 | 主数据、流程数据、执行记录和组织权限沉淀 | 把一次异常决策过程结构化、可解释、可审计 |
| AI 平台 | 企业级 AI 平台、低代码、生态集成和管理助手 | 高风险工业场景里的受控 Workflow、质量门和责任边界 |
| 项目落地 | 长期主系统建设和深度流程改造 | 4-6 周只读 replay / shadow mode 验证增量价值 |

因此，如果客户已经深度使用金蝶，并且现有模块已经覆盖异常识别、影响分析、Top-K 可行方案、计划员确认、审计追溯和执行反馈闭环，ReOrch 不应作为替代方案销售。更现实的定位是行业原型、插件化能力、异常决策验证包，或者接入金蝶数据后的决策增强层。

## 首期场景

ReOrch 的第一商业切口应收敛为：关键瓶颈设备故障或停机导致急单延期风险时，辅助计划员完成局部重排决策。

这个选择比同时覆盖设备故障、插单、物料延期和质量返工更可靠。设备故障有明确触发源，可以来自 MES、IoT、人工报修或设备状态；影响范围可以根据排程快照、设备能力、工序顺序和交期推导；ROI 也更容易解释为人工试排时间、延期分钟、换线次数和协调轮次的变化。当前 demo、数字孪生样例和 `M-03` 故障场景也已经围绕这个方向形成证据链。

插单、物料延期和质量返工不是不重要，而是每一类都需要不同的数据和组织协同方式。它们适合作为后续扩展场景，不适合作为首个对外商业切口。

## 价值验证

ReOrch 当前的 ROI 证据主要来自数字孪生代理结果，不能直接写成客户现场收益。更稳妥的表达是：数字孪生已经证明系统具备记录和计算价值的结构，后续必须用实验室反馈、历史异常 replay、shadow mode 和客户财务口径去复核。

如果计划员不采纳系统推荐，不能简单归因成“系统错了”或“用户保守”。不采纳本身是关键样本，应该被拆成三类：

| 归因 | 典型现象 | 后续处理 |
| --- | --- | --- |
| 数据问题 | 设备状态、加工时长、工艺路线、日历、物料或优先级与现场不一致 | 标记为数据就绪问题，不纳入 ROI 成功结论 |
| 策略/求解问题 | Top-K 没覆盖计划员可接受方案，目标权重与现场偏好冲突，扰动过大 | 调整权重、阈值和策略库，沉淀失败样本 |
| 组织流程问题 | 计划员基于销售承诺、客户关系、班组能力、领导偏好等系统外因素决策 | 把 override 原因结构化，形成约束候选或审批规则 |

真正有说服力的试点，不是只展示一次成功推荐，而是能解释为什么某些推荐被驳回，以及系统如何从这些驳回中校准数据、约束、阈值和组织流程。

## AI 增量

当前默认 demo 仍关闭外部 LLM，且高风险排程、质量门、确认和回写不交给模型，所以不能宣称已经完成真实 LLM 生产验证。更准确的表达是：ReOrch 先用确定性服务和求解器跑通高风险工业决策闭环，并把异常理解、规则候选和计划员反馈结构化接成可配置真实 LLM Agent。

后续 AI 的增量不在“让大模型直接排产”，而在这些更适合模型介入的位置：

- 把自然语言异常、维修描述、MES 告警转为结构化 Incident。
- 把计划员口头规则、override 原因转为待审核 constraint candidate。
- 基于结构化 KPI、质量门和 source refs 生成可审计解释。
- 把每次选择、驳回和执行反馈沉淀成案例资产。
- 在客户历史样本足够后，辅助学习偏好、阈值和策略排序。

五个 Agent 的具体职责、输入输出和质量门见 [ai_increment_agent_design.md](ai_increment_agent_design.md)。这也是作品集应该强调的主叙事：不是“我做了一个 AI 排产系统”，而是“我知道如何把高风险 AI 能力放进可控、可验证、可审计的工业决策流程”。

## 质量门与采纳率

质量门天然会影响采纳率。只会阻断的质量门会让计划员觉得系统不可用；过于宽松的质量门又会把错误方案推向生产。ReOrch 的平衡方式不是取消质量门，而是把它分成 `block`、`warning` 和 `pass` 三层。

`block` 只用于硬约束失败、关键数据缺失、越权回写等不可接受风险；`warning` 允许进入候选列表，但必须解释风险且不能自动预选；`pass` 也不代表自动执行，仍需要计划员确认。这样计划员看到的是 Top-K 方案和取舍，而不是一个不容讨论的“系统最优”。

当前边界也要说清楚：MVP 只能证明质量门机制存在，不能证明阈值已经适配真实客户。阈值必须通过历史 replay、实验室反馈和客户现场 shadow mode 校准。

## PoC 边界

PoC 的北极星指标建议定义为“受控异常决策时间”：从异常确认到计划员完成采纳或驳回一个可审计 Top-K 方案的时间。这个指标不能单独优化，必须有护栏：硬约束可行率、审计完整率、越权写回次数优先级最高；然后才是交付风险、扰动范围、换线次数和采纳率。

20-30 万元付费 PoC、30-50 万元/车间/年的数字只能作为商业假设。它需要通过计划员时间节省、延期和罚金减少、换线和扰动成本下降、审计和关键人依赖降低来验证。完成 10-30 条历史异常 replay、shadow mode 与计划员并行对比、财务口径确认后，才适合升级为报价依据。

数据质量差时，ReOrch 不应硬做成 AI 效果验证。最低门槛是能提供工单、工序、设备、工艺顺序、加工时长、当前排程快照、设备状态或异常事件、交期和优先级，以及至少 10 条可复盘历史异常。若无法重建基准排程或追溯异常影响范围，最小交付物应退回到数据就绪评估和异常决策可行性报告，而不是承诺重排效果。
