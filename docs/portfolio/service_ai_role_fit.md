# 服务领域 AI 产品能力映射

## 1. 定位

ReOrch 的主场景是工业异常调度，但作品集证明的底层能力不是“制造业专属排产”，而是高约束服务流程中的 AI 决策辅助能力：

```text
行业调研 -> 用户需求 -> PRD -> 原型与研发对齐 -> 联调验收
-> 上线指标 -> 反馈复盘 -> Agent / Workflow 迭代
```

该能力可迁移到智能客服、服务工单、售后质检、客户成功、服务运营和行业知识助手等场景。关键不是让模型自由决策，而是把 AI 放进可控、可评测、可追踪的人机协同流程。

## 2. 岗位能力证据

| 能力要求 | ReOrch 中的证据 | 服务领域迁移方式 |
| --- | --- | --- |
| 行业调研 | `market_benchmark.md`、`kingdee_positioning_note.md` 对制造数字化、主系统边界和竞争格局做了收敛 | 对智能客服、工单平台、质检系统、CRM/客户成功工具做同类对标，明确 AI 的补位而非替代范围 |
| 用户需求分析 | `portfolio_proof_matrix.md`、`business_process_flow.md` 梳理计划员、生产主管、IT、质量/审计角色痛点 | 映射为一线客服、班长、质检、运营、知识库管理员、系统集成负责人的流程痛点 |
| PRD 与技术文档 | `prd_decision_workbench.md`、`prototype_logic.md`、`workflow_prompts_io.md` 覆盖用户故事、页面流程、输入输出和 AI 约束 | 可转成客服 Copilot PRD、工单 Agent PRD、服务质检 PRD，并保留降级、权限和验收标准 |
| 项目推进 | `mvp_delivery_plan.md` 将需求、PRD、原型、开发、联调、灰度、指标看板和风险清单拆成里程碑 | 服务 AI 项目同样可按 discovery、MVP、pilot、shadow、controlled rollout 推进 |
| 上线指标 | `metric_system.md`、`launch_readiness_assessment.md` 定义闭环率、采纳率、风险阻断、审计完整率等指标 | 迁移为回复采纳率、转人工率、一次解决率、SLA 风险识别、质检命中率、知识引用覆盖率 |
| 创新输入 | Agent workflow、Data Readiness、Evidence Center、NGS Lab、failure cases 展示了 AI Native 产品创新 | 服务场景可沉淀为 AIGC 回复建议、知识证据层、工单补全、质检 Agent、服务失败样本库 |

## 3. 服务 AI 的迁移逻辑

| ReOrch 能力 | 服务 AI 对应能力 |
| --- | --- |
| Data Readiness Gate | 会话、客户、订单、知识库、历史工单的数据完整性检查 |
| Evidence Stack | 知识库引用、历史案例、政策条款、客户上下文和操作记录 |
| Agent Workflow | 意图识别、信息补全、知识检索、回复建议、风险检查、转人工 |
| Guardrail | 禁止编造政策、禁止越权承诺、低置信转人工、敏感问题升级 |
| Human Confirmation | 坐席采纳、班长审核、质检复核、运营发布 |
| Audit / Case Memory | 回复引用、采纳/驳回、处理结果、失败样本和知识库迭代 |

## 4. 示例场景一：AI 客服 Copilot

| 模块 | 产品设计 |
| --- | --- |
| 用户 | 一线客服、客服主管、知识库运营 |
| 原始痛点 | 知识库分散、回复质量不稳定、复杂问题需要反复问同事、事后难复盘 |
| AI 输入 | 用户问题、客户等级、订单状态、历史会话、知识库片段、服务政策 |
| Agent 流程 | 意图识别 -> 信息补全 -> RAG 检索 -> 回复草案 -> 置信度与风险检查 -> 坐席确认 |
| Guardrail | 低置信不自动回复；政策、退款、投诉升级必须引用知识或转人工 |
| 上线指标 | 回复建议采纳率、平均处理时长、一次解决率、转人工率、错误回复率、知识引用覆盖率 |

## 5. 示例场景二：智能工单 Agent

| 模块 | 产品设计 |
| --- | --- |
| 用户 | 服务运营、工单调度、技术支持、客户成功 |
| 原始痛点 | 工单分类不准、字段缺失、SLA 风险发现晚、处理结果难沉淀 |
| AI 输入 | 工单文本、客户信息、产品版本、历史案例、SLA 规则、服务资源 |
| Agent 流程 | 工单分类 -> 字段补全 -> 风险评分 -> 派单建议 -> 人工确认 -> 处理结果复盘 |
| Guardrail | 关键字段缺失时不派单；高价值客户、投诉、合规问题强制升级 |
| 上线指标 | 首次分类准确率、补全率、SLA 风险提前量、派单采纳率、重开率、处理闭环率 |

## 6. 能力结论

ReOrch 已经证明的不是单一行业 demo，而是一套可迁移的 AI 产品方法：先做行业与用户问题收敛，再写可交付 PRD 和原型逻辑，用 Agent/Workflow 补足高认知负担环节，通过质量门、人工确认、上线指标和失败样本库控制不确定性。服务领域 AI 产品可以复用这套方法，但必须重新校准知识库、权限、SLA、用户体验和业务指标。
