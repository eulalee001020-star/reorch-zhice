# 当前项目评审

本文按三个问题评审 ReOrch 智策当前实现：结果是否可信可行、如何控制 AI 成本、商业价值是否成立。结论先行：项目已经达到可互动 PoC 作品集标准，但还不能被描述为真实客户生产验证完成；下一阶段应进入历史 replay、shadow mode 和客户阈值校准。

## 1. LLM 结果是否可信可行

ReOrch 当前不是让 LLM 直接给排产答案，而是把 Agent 输出放进确定性验证链路。异常理解、策略组织和解释可以由 Agent 参与；影响计算、约束校验、候选方案、推荐排序和回写必须由结构化模型、求解器、质量门和人工确认控制。

| 判断项 | 当前状态 | 证据 | 提升空间 |
| --- | --- | --- | --- |
| 结构合法 | 已实现 | Pydantic models、TypeScript types、demo mapping validation | 接客户数据时补字段血缘和版本管理 |
| 数据可追溯 | 部分实现 | Incident、ScheduleSnapshot、ImpactReport、AgentTraceStep 已串联 | 解释层需要逐条 source_refs，精确到工单、工序、设备、KPI |
| 硬约束 | 已实现核心版 | ConstraintValidator、HybridSolver、PlanRecommendationEngine、PlanQualityGate | BOM、人员、刀具、能耗等客户约束需要按现场数据补齐 |
| 业务风险 | 已实现核心版 | risk_flags、quality_gates、recommendation_explanation、前端候选方案质量门标签 | 风险阈值需要客户历史异常校准 |
| 置信度 | 已实现 | Incident/Strategy/Recommendation confidence，低置信度不自动预选 | 需要记录人工覆盖后的置信度校准样本 |
| 审计 | 核心链路已实现 | DecisionRecord、audit_metadata、writeback records、AgentTraceStep | 生产级审计包导出、模型/规则版本快照仍需增强 |

当前已补齐：异常决策主流程返回 `quality_gates`，并在候选方案比较表显示“质量门通过/预警/阻断”。这让质量门不只停留在文档和初始排产流程，而是进入核心异常工作台。

## 2. 成本与模型选择

当前 demo 没有接入外部 LLM 供应商，因此不存在真实 token 账单。这个状态适合作品集展示，因为高风险工业决策的正确性主要由确定性工具承担；但如果进入商业 PoC，需要补模型路由、token 计量和按任务降级策略。

| 模块 | 当前实现 | 推荐模型/工具策略 | 成本判断标准 |
| --- | --- | --- | --- |
| 数据导入与字段映射 | 规则校验 + schema | 规则优先，小模型只做字段名候选 | schema error rate、人工修正字段数 |
| 异常文本理解 | 规则/正则 + 置信度 | 小模型或轻量分类器即可，低置信度人工确认 | 准确率、低置信召回、单次成本 |
| 影响分析 | 确定性引擎 | 不使用 LLM | 计算耗时、影响范围准确率 |
| 候选方案生成 | OR-Tools/求解器 | 不使用 LLM | 可行率、P95 求解耗时 |
| 方案解释 | 模板化解释层 | 中小模型可润色，但必须引用结构化证据 | 幻觉率、source_refs 覆盖率、人工采纳率 |
| 案例沉淀/反馈结构化 | 结构化抽取 | 小模型可用，输出必须 schema 校验 | 可复用规则比例、人工审核通过率 |

成本控制的原则是：能用规则和求解器解决的，不交给大模型；能用小模型稳定完成的，不调用大模型；模型只负责语义、总结、解释和经验抽取，不负责最终可行性判断。

[补充] 进入真实 PoC 前应补三类成本观测：每次 workflow 的模型调用次数、input/output token、以及按模块分摊的单次异常处理成本。上线阈值建议是：高频异常的单次 AI 成本必须显著低于计划员节省时间价值，并且不得因为解释润色消耗主要预算。

## 3. 商业化价值判断

项目价值不应靠“AI 排产”概念成立，而应靠异常响应速度、交付风险降低、计划员决策负担下降和经验资产沉淀成立。当前项目已经有 PoC 路径、ROI 测算、价值仪表盘和 demo 数据，但还没有真实客户生产数据验证。

| 维度 | 当前状态 | 证据 | 仍需验证 |
| --- | --- | --- | --- |
| 场景痛点 | 已定义清楚 | 设备故障等异常触发的重排决策 | 客户现场异常频率和损失分布 |
| 用户角色 | 已定义 | 计划员、主管、IT/集成、执行端 | 真实计划员工作流访谈 |
| 产品闭环 | 已跑通 PoC 版 | 异常 -> 影响 -> 方案 -> 推荐 -> 确认 -> 回写 -> 案例 | shadow mode 下与人工方案对比 |
| 价值测算 | 已有工具 | ValueTrackingService、PoC Dashboard | 客户财务口径和实际单价 |
| 付费路径 | 已设计 | 4-6 周付费 PoC、单车间切入 | 愿付费访谈和采购阻力 |
| 风险边界 | 已明确 | 只读接入、人工确认、mock writeback | 真实系统适配和回写失败处置 |

判断是否不是自嗨式开发，不能只看 demo 是否完整，而要看是否满足以下条件：

| 判断标准 | 当前结论 |
| --- | --- |
| 是否绑定高频、高损失、有人负责的业务事件 | 部分成立，需客户异常频率数据确认 |
| 是否比现有人工流程更快、更稳或更可追责 | PoC 逻辑成立，需 shadow mode 数据验证 |
| 是否能在不替换 APS/MES 的前提下上线 | 架构成立，需客户接口验证 |
| 是否有明确停损标准 | 已在市场对标文档中定义，但需项目管理执行 |
| 是否有愿付费主体和预算口径 | 方向成立，仍需真实客户访谈和报价验证 |

## 4. 当前补齐后的结论

| 层级 | 结论 |
| --- | --- |
| 作品集展示 | 已具备：GitHub、可互动 demo、工作流、prompt、输入输出、质量门、市场对标 |
| 技术 PoC | 基本具备：后端测试、前端构建、demo 数据校验、核心异常闭环 |
| 工业可信性 | PoC 级成立：硬约束、质量门、人工确认、审计链路已经有代码证据 |
| 成本治理 | 设计清楚，但真实模型路由和 token telemetry 未落地 |
| 商业验证 | 逻辑清楚，但还缺真实客户 replay、shadow mode、WTP 和财务口径验证 |

## 5. 下一阶段优先级

| 优先级 | 工作 | 验收标准 |
| --- | --- | --- |
| P0 | 补 source_refs 到解释层 | 每个核心结论能追溯到数据对象或约束报告 |
| P0 | 做历史 replay 数据集 | Top-N 包含人工方案比例、方案可行率、响应时间可量化 |
| P1 | 建模型路由与 token telemetry | 每个 Agent step 有模型、token、成本、latency 记录 |
| P1 | 客户阈值校准 | 延期、换线、扰动、瓶颈风险阈值可配置 |
| P1 | 审计包导出 | 一次决策可导出输入、输出、版本、确认、回写和执行反馈 |
